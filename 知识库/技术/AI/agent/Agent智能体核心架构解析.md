# Agent 智能体核心架构解析：从 Workflow 到 ReAct 推理

> 本文基于 Agent 理论课程学习笔记整理，系统梳理 Workflow Agent、ReAct Agent 及 Agent 平台的核心概念与设计思想。

---

## 一、为什么需要 Agent？

大语言模型（LLM）的能力边界很清晰：**会说，但不一定能做；能规划，但不一定靠谱**。幻觉问题、执行断层、规划失控，这些都是纯 LLM 方案的硬伤。

Agent（智能体）的本质，就是在 LLM 外部包一层**控制与执行框架**，让模型的"智力"能够可靠地转化为"结果"。

一句话概括：**AI 模型提供智能，AI Agent 保证结果。**

---

## 二、Workflow Agent：工作流智能体

### 2.1 核心思想

Workflow Agent 是一种**白盒化**的方案——提前把任务拆成固定步骤，LLM 在每个节点按需被调用，整体流转由代码逻辑严格控制。

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  输入   │───→│ 步骤 1  │───→│ 步骤 2  │───→│  输出   │
│ (Prompt)│    │(LLM调用)│    │(工具调用)│    │(结果聚合)│
└─────────┘    └─────────┘    └─────────┘    └─────────┘
         ↑ 每个步骤的输出作为下一步的输入，流程固定
```

### 2.2 解决什么问题？

| LLM 的局限 | Workflow Agent 的应对 |
|-----------|---------------------|
| 幻觉 | 关键节点用代码校验，不依赖模型自判 |
| 只会说不会做 | 每个步骤绑定具体工具/插件执行 |
| 规划不靠谱 | 流程预定义，模型只做"填空" |

### 2.3 适用场景

- 标准化业务流程（如审批、填报、数据清洗）
- 对准确性要求极高的场景（如医疗、金融）
- 需要审计回查的任务（白盒可追溯）

### 2.4 关键权衡

Workflow Agent 的核心设计问题是：**到底给 LLM 多大的自由空间？**

- 自由度太低 → 模型变成模板填充器，浪费智能
- 自由度太高 → 又回到不可控的老路

最佳实践是：**在"必须对"的节点用代码兜底，在"可以智"的节点给模型空间。**

## 三、ReAct Agent：推理型智能体

### 3.1 核心思想

有些任务**无法提前设定步骤**，必须让模型在运行时根据环境反馈动态决策。这就是 ReAct（Reasoning + Acting）的核心：**推理 → 行动 → 观察反馈 → 再推理，循环往复。**

```
┌─────────────────────────────────────────────┐
│  Question: 上海明天的天气适合户外跑步吗？      │
└─────────────────────────────────────────────┘
                       ↓
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Thought  │───→│  Action  │───→│Observation│
│ "需要查   │    │ 调用天气  │    │ "明天晴，  │
│  天气"    │    │ API      │    │  25°C"    │
└──────────┘    └──────────┘    └────┬─────┘
      ↑                              │
      └──────────────────────────────┘
               (循环，直到得出最终结论)
```

### 3.2 每一轮的本质

一个容易被忽略但极其关键的洞察：

> **AI 的所有动作，本质上都是为了拼出信息更完整的 Prompt，以获得最好的结果。**

每一轮的 Observation（观察结果）被拼接到历史上下文中，模型的推理基础就越来越扎实。ReAct 不是"让模型变聪明"，而是**让模型在更完整的信息上做决策**。

### 3.3 与思维链（Chain of Thought, CoT）的关系

| 概念 | 作用 | 与 ReAct 的关系 |
|-----|------|---------------|
| CoT | 让模型"想一会再回答"，输出中间推理过程 | ReAct 的 Thought 环节就是 CoT 思想的工程化落地 |
| ReAct | 不仅要想，还要**做**，根据反馈调整 | CoT 是静态推理，ReAct 是动态交互 |

**实践提示**：即使是单轮问答，加上"Let's think step by step"（CoT 提示）往往也能显著提升准确率。

### 3.4 工程实现要点

1. **最大循环轮次必须设限** — 防止无限循环
2. **Finish Action** — 模型需要有能力自行判断任务完成并输出终止信号
3. **上下文压缩（待深入研究）** — 如果问题特别复杂，多轮拼接后的 Prompt 会很快触及 Token 上限。工业级实现中是否做了总结/截断/摘要优化？这是一个值得深挖的工程细节。

### 3.5 模块化理解：Action = 子 Agent

一个关键的抽象升级：

> ReAct 中每一轮执行的 Action，可以看作是一个具备特定能力的**子 Agent**被调用。

整个 ReAct 流程就是**主 Agent 动态编排多个子 Agent**的过程。这与 Multi-Agent 架构的思想一脉相承。

## 四、关键概念澄清：MCP、Function Call、Skill

在 Agent 体系中，这几个词经常混用，有必要厘清：

| 概念 | 全称/含义 | 定位 |
|-----|----------|------|
| **MCP** | Model Context Protocol（模型上下文协议） | **调用协议层** — 统一不同工具/数据源的接入标准 |
| **Function Call** | 函数调用 | **能力接口层** — 模型输出结构化指令，由框架解析并执行具体函数 |
| **Skill** | 技能 | **能力实现层** — 具体的工具、插件、工作流等业务能力 |

**关系**：MCP 解决"怎么连"，Function Call 解决"怎么调"，Skill 解决"做什么"。

---

## 五、Agent 平台

Agent 平台是承载上述所有能力的**基础设施层**，核心职责：

- **管理**：Agent 注册、版本、权限、生命周期
- **运行**：调度执行、资源隔离、超时熔断
- **支持**：工具市场、数据接入、监控观测、日志回溯

可以类比为：Agent 是应用，Agent 平台是操作系统 + 应用商店 + 运维体系。

## 六、Workflow vs ReAct：怎么选？

| 维度 | Workflow Agent | ReAct Agent |
|-----|---------------|-------------|
| 可控性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 灵活性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 适用任务 | 步骤明确、可预设 | 路径多变、需探索 |
| 调试难度 | 低（白盒） | 高（黑盒/灰盒） |
| 典型场景 | 审批流、报表生成 | 复杂问答、研究调研 |

**混合策略**：在实际工程中，往往是两者结合——用 Workflow 保证主流程不跑偏，在特定节点嵌入 ReAct 处理不确定环节。

## 七、总结与个人思考

1. **Agent 不是替代 LLM，而是给 LLM 装上手和眼** — 模型负责想，Agent 框架负责做和校验。

2. **ReAct 的"循环"思想值得推广** — 不仅限于 Agent，任何需要多步决策的系统（如投资分析、代码 review）都可以借鉴"推理-行动-反馈-再推理"的闭环。

3. **上下文管理是工业级 Agent 的隐形战场** — 课程中提到的 Token 上限问题，在实际落地时大概率需要引入摘要、截断、分层记忆等机制，这是从 Demo 到生产的关键一跃。

4. **MCP 的意义被低估** — 统一协议层的出现，意味着 Agent 生态可能迎来类似"HTTP 之于互联网"的标准化时刻，值得关注其演进。
