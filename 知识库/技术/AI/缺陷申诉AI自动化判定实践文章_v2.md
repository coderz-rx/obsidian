# 规则未尽之处，AI如何给出确定性判定：从0到1构建高确定性AI业务判定系统

## 一、项目背景

我们是OTA行业酒店售卖业务的技术团队。日常运营中，酒店违约（拒单、涨价等）后，代理商可以申诉，由运营人员审核决定是否撤销惩罚。

这个流程长期存在两个结构性问题：

| 问题 | 具体表现 |
|------|---------|
| **判责不一致** | "具体情况具体分析"没有标准化流程，不同运营人员可能给出相反结论 |
| **知识未沉淀** | 有经验的运营脑子里装着大量隐性规则，离职后跟着人走 |

每天300-400个申诉case，4个全职运营在订单系统、客服系统、日志系统、申诉系统之间来回跳转收集信息，靠人工经验做判定。AI能力已经足够成熟，我们判断：**这件事，值得用AI重做一遍**。

---

## 二、从0到1的落地实践：验证→踩坑→解决→上线

### 2.1 快速验证：3天MVP告诉我们"方向对，但不能想得太简单"

我们没有直接立项做正式系统，先用3天搭了一个MVP：手动输入订单信息、客服记录、申诉材料，调用大模型按现有规则文档做判定，跑了50个简单场景的case。**结果：60%一致率。**

不高，但足够让我们看到希望——模型对规则的理解、对信息的提取总结，方向是对的。让我们正式投入的信号不是数字，而是**模型输出的逻辑链条让我们觉得"这事可行"**。然而，MVP也提前暴露了坑：我们以为"按规则文档判定就行"，但50个case里已经有好几个是规则文档覆盖不全的，模型按显性规则给出判定，运营的实际判定却完全相反。**MVP的最大价值：提前告诉我们，真正的难点不在技术，在业务规则的完整性——文档写了的不一定就是全部。**

### 2.2 正式落地启动：数据整合与流程设计

验证可行后，我们开始正式搭建系统。**第一步，整合数据。** 原本运营处理一个case要在4个系统间来回跳转，我们先做了数据聚合，搭建申诉判定工具平台，让信息一目了然，这一步纯工程，相对顺畅。**第二步，设计自动化流程。** 看起来是pipeline工程，但真正的难点在规则怎么写：获取订单信息→提取客服记录关键信息→提取订单日志（进单、拒单等商家动作）→提取代理商提交的凭证（图片/录音/文字）→所有信息打包提交给大模型，按规则判定。**技术栈**：文本提取与判定用GPT-4o mini，图片信息提取用阿里云Qwen-VL-Max。

### 2.3 落地中的坑与解决：我们以为在做技术，其实是在做"业务考古"

#### 坑一："统一大模型"的幻想破灭

最初我们想得很优雅：一个通用大模型 + RAG规则库，覆盖所有申诉场景。结果测试一致率：**51.34%**。问题出在两个层面：一是信息提取层面，想让提取规则覆盖所有场景就必然导致规则泛化，解决了A场景，B场景的信息提取效果又崩了；二是判定层面，不同场景的判定细节差异巨大，统一模型试图涵盖所有细节，结果就是提示词内容爆炸，模型注意力涣散。**关键教训：追求"统一"是技术人的本能，但业务场景的多样性会惩罚这种本能。**

**解法：场景拆分——从"大一统"到"分而治之"**。我们改变了策略，按申诉场景拆分，每个场景给独立的信息提取规则和判定规则，效果立竿见影——**一致率从51.34%提升到78.6%**。对于还没拆分的零散场景，继续走"兜底模型"流程，区别在于已经拆分场景的case被剥离后，兜底模型的注意力更集中了。

#### 坑二：申诉材料的"七十二变"

代理商提交的凭证材料，形式和内容完全不统一——微信聊天截图、政府征用公告、公众号通知、工人修电路的照片、酒店系统后台截图、电力部门系统截图……我们原本想制定"固定图片信息提取规则"，结果发现这是妄念。**材料形态的多样性，本质是业务不确定性的外化**，只能让多模态模型（Qwen-VL）尽量提取关键信息，然后在判定规则里留足弹性空间。

#### 坑三：规则只写了"显性层"，真正的判定逻辑藏在"隐性层"

这是最深、最隐蔽的一个坑，也是整个项目最大的认知收获。

**case：酒店火灾。** 代理商提交申诉："酒店凌晨发生火灾，部分房屋受损无法办理入住，申请撤销拒单惩罚。"AI读完规则文档，给出了**"申诉通过"**的判定——规则里写着"不可抗力场景，提供官方说明即可通过"。但运营看了，直接点了**"不通过"**："等等，这条得先看火灾是什么原因导致的。如果是他们自己线路老化引起的，那是酒店责任，不能算不可抗力。"

问题出在哪儿？规则文档只写了第一层判定：有官方证明即可。但它没写第二层判定：火灾原因是否属于酒店自身责任。这条规则写在哪儿？答案是——**没有写在任何文档里**。它存在于某个资深运营的大脑里，存在于无数次申诉审核后的经验中。

**这种case怎么发现的？** 不是靠运营"教"我们，而是靠AI跑完case后跟人工结果做diff，从差异case里反向挖掘的。AI按文档判定"通过"，人工判定"不通过"——差异本身就是一条未记录的规则。运营很难通过人脑把隐性经验完整总结成文档，但AI可以持续产出结果，我们持续对比，**从"AI错在哪"里提炼出"原来还有第二层判定"**。这个过程本质上是在做**业务考古**——把藏在人脑里的**规则分层逻辑**，一点一点挖出来，写成显性规则，喂给AI。

现在，这条规则已经写进了我们的判定规则库，而且是**分层结构**的：第一层判定是否属于不可抗力场景（是否有官方证明）；第二层，如果涉及火灾/爆炸等事故，进一步判定事故原因是否属于酒店自身责任（如线路老化、管理疏忽）。

### 2.4 控制影响范围的上线策略：双跑灰度，对业务结果负责

申诉判定直接影响线上业务（代理商是否被惩罚），不能草率上线。我们采用**两阶段开量**：阶段一，双跑灰度——AI判定和人工判定同时进行，对比结果差异，持续优化提示词和规则，目标一致率稳定超过85%；阶段二，正式上线——当前已上线1个细分场景，上线后一致率**超过85%**，预期全部场景上线后人力从4人降至1人。

**核心原则：AI项目的上线标准不是"模型准确率够高"，而是"对业务结果的影响可控"。** 双跑灰度的真正价值，不只是验证准确率，更是**用每天的对比数据积累信任**——让运营亲眼看到"这单AI判得和我一样"，比任何技术参数都更有说服力。

## 三、总结：经验、建议与反思

### 3.1 核心感悟：AI落地的最大阻力不是模型能力，而是"模糊输入"与"确定输出"的矛盾

做完这个项目，我最深的感悟是：**AI项目落地最大的阻力，往往不是模型能力不够，而是"输入模糊"与"输出必须精准"之间的矛盾。**

大模型本质是概率模型，最擅长在模糊信息里找"最可能的答案"。但如果业务对结果精准度要求极高（比如申诉判定直接影响代理商是否被处罚，错了就是业务风险），概率模型的天然属性和业务需求就会产生直接冲突。我们面对的场景恰恰就是这个矛盾的经典组合：规则不够清晰，大量判定逻辑藏在运营的经验里，没写在任何文档中；精准度要求极高，判定结果直接影响线上业务，不允许"大概率对就行"。

我们化解这个矛盾，靠的是两条路径并行。

**路径一：工程兜底，降低精准度要求对系统的压力。** 规则引擎保确定性（边界清晰的场景直接走规则），大模型处理模糊场景（材料理解、语义提取、辅助推理），灰度双跑 + 人工兜底，把"必须100%准确"转化为"风险可控、逐步放量"——本质上是用工程架构弥补模型在确定性上的不足。

**路径二：业务考古，把模糊性逐步转化为清晰度。** 不从文档出发，从"AI vs 人工的diff"反向挖掘规则，把隐性经验显性化、分层化（就像火灾case里挖出的"第二层判定"）。前期花的最多时间不是写代码，而是和运营一次次对case、一次次厘清"这种情况下你们到底怎么判"。

> **前期业务层面的工作做得越充足，后期技术落地就越顺畅。** 如果运营能在一开始给出覆盖所有场景的具体规则细节，技术推进会流畅得多；反之，如果业务本身就带有模糊性，项目落地就少不了反复确认细节、确认边界。换句话说：**事情越具体，AI项目落地越顺利。**

但这还不是全部。规则不清晰的场景里，除了技术解法，还有一层**组织层面的信任建设**。业务方天然担心"AI不懂那些没说出来的规则"，所以双跑灰度的真正价值不只是验证准确率，更是**用每天的对比数据积累信任**。**技术可以加速转化，但无法替代转化；数据可以积累信任，但无法跳过信任建设的过程。**

### 3.2 给其他AI落地项目的建议

| 阶段 | 建议 |
|------|------|
| **启动前** | 不要假设业务有清晰规则，先花时间和业务方共同梳理，把隐性知识显性化；**同时评估业务对结果精准度的容忍度**——容错率极低的场景需要更重的工程兜底设计 |
| **验证期** | 用MVP快速验证可行性，但更重要的是用MVP暴露"规则盲区" |
| **开发期** | 警惕"统一大模型"的诱惑，场景拆分往往是更好的策略 |
| **迭代期** | 用"AI vs 人工差异case"反向挖掘规则，比让业务方"总结规则"更有效 |
| **上线期** | 采用双跑灰度，对业务结果负责，不要只看模型指标 |

### 3.3 写在最后

回到那个火灾case。AI现在遇到这种case，会先判断第一层，再判断第二层，给出和运营一样的结论。但这个case的价值不止于此——它提醒我们：**AI落地的过程，本质上是一个把"人的经验"转化为"系统的规则"的过程。而经验的隐性分层（规则不止一层，深层规则没写在文档里），是这个转化中最隐蔽的阻力。技术可以加速这个转化，但不能替代这个转化。**

这是我们花3个月、处理数百个边界case、把一致率从51%提升到85%的过程中，学到的最重要的一课。
