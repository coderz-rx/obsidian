# 酒店缺陷申诉AI自动化判定实践：年省750人天的智能化升级之路

> 作者：张瑞兴 | 去哪儿网酒店供应链技术团队

## 一、业务背景与痛点分析

在去哪儿网酒店业务中，订单缺陷管理是保障服务质量的关键环节。当用户投诉订单被平台标记为缺陷单并处罚代理商后，代理商有权发起申诉请求撤销判定。

### 1.1 业务规模与人力成本

**日均申诉量：**
- C直采：约300单/天
- 旗舰店+Q代理：约50单/天
- **合计：约350单/天**

**人力投入现状：**
- 运营审核团队：4人天/天
- 按每年250个工作日计算：**年投入1000人天**
- 单均处理时长：**5分钟/单**

### 1.2 现有流程的三大痛点

**痛点一：处理效率低下**

传统人工处理流程繁琐，审核同学需要：
- 登录多个系统查询订单详情、用户反馈、服务记录
- 查看申诉材料（可能涉及图片、语音、文字描述）
- 对照判定规则逐条核验
- 单case平均耗时5分钟，高峰期积压严重

**痛点二：判责主观性强，缺乏一致性**

遇到"具体情况具体分析"的场景时：
- 缺乏标准化流程，依赖个人经验决策
- 不同审核人员对同类case可能给出不同判定结果
- 判定结果难以追溯和解释

**痛点三：业务知识未体系化沉淀**
- 有经验员工的判定逻辑分散在个人头脑中
- 新人培训成本高，上手周期长
- 知识难以传承，人员流动带来业务风险

### 1.3 项目目标

通过AI大模型自动完成缺陷申诉的判定，实现：
- **效率提升：** 减少人力投入，提高申诉判定效率
- **质量保障：** 解决判责主观性问题，保证一致性
- **知识沉淀：** 将经验转化为可传承的组织资产

## 三、项目实施过程

### 3.1 业务调研：明确"干什么、怎么干"

**核心任务：** 确认运营如何完成申诉判定，包括完整流程和每个步骤的具体细节。

**输出成果：**
1. **业务流程图：** 完整的申诉判定流程图，覆盖从工单接收→材料收集→规则匹配→判定输出→结果反馈的全链路
2. **技术能力需求指标：**

| 能力需求 | 具体说明 | 业务占比 |
|---------|---------|---------|
| 视频理解与内容总结 | 处理用户上传的投诉视频 | 低（暂不覆盖） |
| 图片理解与内容总结 | 识别图片中的文字、场景、凭证 | 中 |
| 语音理解与内容总结 | 处理语音投诉、电话录音 | 中 |
| 判定规则检索与综合判定 | 基于规则库进行逻辑推理 | 高（核心能力） |

### 3.2 技术调研：判定落地可行性

针对各项技术能力需求，我们进行了详细的技术可行性调研：

| 能力需求 | 技术实现路径 | 可行性结论 |
|---------|-------------|-----------|
| **视频理解** | 多模态大模型 | ❌ 当时业内无成熟方案，Q技术平台暂不支持 |
| **图片理解** | 多模态大模型（OCR+文本理解） | ✅ Q平台已接入支持图片识别的多模态大模型 |
| **语音理解** | ASR转文本 + 文本大模型理解 | ✅ Q技术平台可提供ASR能力，文本大模型已接入 |
| **规则检索与判定** | Dify知识库外挂 + qmoss Agent能力 | ✅ Q平台提供Dify知识库和Agent能力支持 |

**关键决策：** 由于视频处理能力尚不成熟，且涉及视频文件的申诉样本占比较低，我们决定MVP阶段暂不覆盖这部分case，优先攻克图片、语音和规则判定能力。

### 3.3 MVP方案设计与落地

#### 3.3.1 数据准备

"数据是模型的粮食"，我们在MVP第一阶段优先完成数据准备工作：

**数据整合：**
- 申诉数据、客服信息、订单日志分散于各个系统，需要统一汇总
- C直采缺陷数据存储于C平台，无法通过系统接口同步，需要开发数据上传工具

**数据质量保障：**
- 建立数据清洗流程，处理缺失值、异常值
- 构建数据标注规范，确保训练数据质量
- 划分训练集、验证集、测试集，避免数据泄露

#### 3.3.2 系统架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                     AI自动化判定系统                          │
├─────────────────────────────────────────────────────────────┤
│  接入层      │  申诉工单接入、材料解析（图片/语音/文本）      │
├──────────────────────────────────────────────────────────────┤
│  预处理层    │  OCR识别、ASR转写、内容提取与结构化          │
├──────────────────────────────────────────────────────────────┤
│  知识检索层  │  Dify知识库规则检索，召回相关判定规则          │
├──────────────────────────────────────────────────────────────┤
│  推理判定层  │  qmoss Agent综合推理，生成判定结论            │
├──────────────────────────────────────────────────────────────┤
│  后处理层    │  置信度评估、结果格式化、人工复核流转          │
└─────────────────────────────────────────────────────────────┘
```

**核心流程：**
1. 接收申诉工单，解析申诉材料类型
2. 图片材料经OCR提取文字，语音材料经ASR转写为文本
3. 将结构化后的case信息送入Dify知识库，检索相关判定规则
4. qmoss Agent结合规则和case上下文进行推理判定
5. 输出判定结果及置信度评分

#### 3.3.3 提示词工程与模型优化

**提示词设计原则：**
- **角色定义：** 明确AI的审核员角色和判定标准
- **Few-shot示例：** 引入典型case的判定示例，提升模型对业务场景的理解
- **规则注入：** 将判定规则动态注入prompt，保证判定的可解释性
- **输出约束：** 规范输出格式，便于结果解析和人工复核

**迭代优化策略：**
- 建立bad case回流机制，人工复核结果自动反哺模型优化
- 定期分析模型误判原因，针对性调整提示词和规则库
- A/B测试验证优化效果，数据驱动迭代

### 3.4 灰度验证与上线保障

**灰度双跑机制：**

在正式接入AI判定能力前，我们设置了严格的双跑验证阶段：
- AI判定结果与人工判定结果并行输出
- 人工审核同学对AI判定结果进行复核
- 监控AI判定的准确率、召回率、一致性指标

**风险控制考量：**

如果AI泛化能力较弱，上线后出现较多错误判定：
- ❌ 会令平台商户对平台能力产生质疑，影响平台形象
- ❌ 出现较多的重复判定，大大增加平台投入成本
- ❌ 可能引发商户投诉，带来业务风险

**上线SOP：**
- 灰度流量控制，逐步提升AI判定占比
- 实时监控模型表现，异常指标自动熔断
- 完备的降级预案，确保人工兜底能力

### 4.2 业务价值体现

**1. 降本增效显著**

原本需要4人天/天的审核工作，现在仅需1人天/天即可完成。审核同学从重复性劳动中解放出来，将精力聚焦于复杂case的深入分析和规则体系的持续优化。

**2. 判定质量提升**

- **一致性：** 消除人为主观差异，同类case判定结果统一
- **可追溯：** 每个判定结果可查看推理依据和引用规则
- **可解释：** 判定逻辑透明，便于向代理商解释说明

**3. 知识资产沉淀**

- 将分散在个人经验中的判定逻辑体系化、数字化
- 新人培训周期从数周缩短至数天
- 人员流动不再带来业务知识流失风险

**4. 用户体验改善**

- 申诉处理时效大幅提升，代理商等待时间缩短
- 判定标准透明化，减少争议和反复申诉
- 运营同学工作满意度提升，专注高价值工作

## 五、经验总结与方法论沉淀

### 5.1 AI项目落地的关键经验

**经验一：业务理解先于技术实现**

我们花了近三分之一的时间在业务调研和规则梳理上。AI不是万能药，深入理解业务场景、梳理清晰的判定规则是成功的基石。技术只是手段，解决业务问题才是目的。

**经验二：MVP思维降低落地风险**

- 不追求完美方案，先验证核心假设
- 视频处理不成熟就先放弃，聚焦图片和语音
- 快速迭代，用数据说话，而非拍脑袋决策

**经验三：人机协作优于全自动**

追求100%自动化既不现实也不必要：
- 置信度高的case由AI自动处理
- 置信度低的case流转人工复核
- 复杂边界case保留人工最终裁决权

**经验四：灰度验证是风险控制的关键**

AI模型的"黑盒"特性决定了上线前必须充分验证：
- 双跑机制确保模型泛化能力
- 异常熔断防止批量错误
- 降级预案保障业务连续性

### 5.2 可复用的方法论

**AI业务应用落地五步法：**

1. **业务调研：** 明确流程、边界、技术能力需求
2. **技术可行性评估：** 调研业界方案，评估平台能力
3. **MVP快速验证：** 聚焦核心场景，快速试错迭代
4. **数据驱动优化：** 分析bad case，针对性调优
5. **灰度验证上线：** 双跑验证，逐步放量，风险可控

**适用场景：** 规则明确、流程固定、数据可获取的业务场景，如审核、客服、风控等。

## 六、未来展望

### 6.1 能力扩展方向

**1. 多模态能力提升**
- 视频理解与内容总结（待技术成熟后接入）
- 更复杂的图片场景识别（如凭证真伪鉴别）
- 多轮对话式申诉处理

**2. 覆盖场景扩展**
- 其他业务线的申诉处理场景（如机票、火车票）
- 客服智能问答与工单分类
- 风险预警与主动拦截

**3. 智能化升级**
- 引入Agentic Workflow，支持复杂场景的多轮推理
- 结合RAG技术，构建更完善的业务知识库
- 探索模型自学习能力，持续优化判定效果

### 6.2 平台建设思路

**构建通用智能审核平台：**
- 规则配置化：业务同学可自助配置判定规则
- 模型可插拔：支持接入不同厂商的大模型能力
- 流程可编排：灵活配置审核流程和人工介入节点
- 效果可观测：完善的效果评估和监控体系

## 七、结语

酒店缺陷申诉AI自动化判定项目从真实的业务痛点出发，通过"业务调研→技术验证→MVP落地→数据优化→灰度上线"的五阶段推进，实现了年省750人天的显著成效。

这一实践验证了我们在AI应用落地中的核心理念：
- **技术服务于业务：** 不做技术炫技，专注解决真实问题
- **数据驱动决策：** 用量化指标衡量效果，而非主观感受
- **风险可控优先：** AI应用必须可解释、可干预、可回退
- **人机协作共赢：** AI处理标准化场景，人工专注复杂决策

未来，我们将继续探索AI在酒店供应链智能化方向的更多可能性，持续为业务创造价值，为用户提供更优质的服务体验。
