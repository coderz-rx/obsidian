## 二、为什么写这篇文章

### 2.1 背景：AI不是洪水猛兽，但也不是万能药

- **乐观派**：认为AI可以替代一切， prompt写得好就能解决所有问题
- **悲观派**：认为AI只是玩具，生产环境不敢用、不能用

真实情况是：**AI在特定场景下可以创造巨大价值，但前提是——你得知道它的能力边界，并为之设计合适的架构。**

### 2.2 冲突：理想（全自动AI判定）vs 现实（遗留系统+复杂业务）

我们面对的不是一张白纸，而是：

- 分散在多个系统的数据
- 对判定准确性要求极高的业务场景

这不是一个"调个API就能搞定"的项目。

### 2.3 文章价值定位

本文面向：
- 想落地AI项目但不知从何下手的技术同学
- 负责业务系统智能化改造的技术负责人
- 对"AI如何与遗留系统结合"感兴趣的同学

你将看到：**从业务调研到灰度上线的完整五阶段实践，以及每个阶段的关键决策依据。**

## 三、技术全景：AI判定系统的能力分层

在深入项目细节前，先建立对"AI自动化判定系统"的整体认知。一个完整的AI判定系统包含四层能力：

```
┌─────────────────────────────────────────────────────────────┐
│  应用层      │  申诉工单接入、结果输出、人工复核界面          │
├──────────────────────────────────────────────────────────────┤
│  推理层      │  大模型推理、规则引擎判定、置信度评估          │
├──────────────────────────────────────────────────────────────┤
│  知识层      │  判定规则库、历史案例库、业务知识图谱          │
├──────────────────────────────────────────────────────────────┤
│  数据层      │  订单数据、客服信息、日志数据、申诉材料          │
└─────────────────────────────────────────────────────────────┘
```

**关键认知：** AI不是替代人工，而是承担"规则明确、流程固定"的标准化判定，复杂case仍需人工兜底。这是"人机协作"而非"全自动"。

---

## 四、问题定义：传统人工判定的三大痛点

### 4.1 效率痛点：流程繁琐，单均5分钟

运营同学处理一单申诉需要：
1. 登录客服系统查看申诉详情
2. 登录订单系统查询订单信息
3. 查看用户反馈、服务记录
4. 对照判定规则逐条核验
5. 做出判定并填写理由

**平均耗时5分钟/单**，高峰期积压严重，影响代理商体验。

### 4.2 一致性痛点："具体情况具体分析"=没有标准

遇到模糊场景时，缺乏标准化流程：
- 不同审核人员对同类case可能给出不同判定
- 判定逻辑依赖个人经验，难以解释和追溯
- 新人上手慢，培训成本高

### 4.3 知识痛点：经验在人脑里，不在系统中

- 有经验员工的判定逻辑分散在个人头脑中
- 人员流动带来业务知识流失风险
- 无法基于历史数据持续优化判定策略

### 4.4 量化影响

| 指标 | 数值 |
|------|------|
| 日均申诉量 | 350单（C直采300+其他50） |
| 人力投入 | 4人天/天 |
| 年人力成本 | 1000人天/年 |
| 单均处理时长 | 5分钟 |

**核心诉求：** 在保证判定质量的前提下，大幅提效降本，并实现知识沉淀。

## 五、解决框架：AI项目落地的五阶段方法论

基于对AI项目的认知，我们制定了"业务调研→技术调研→MVP落地→数据优化→灰度验证"的五阶段推进策略：

```
┌──────────────────────────────────────────────────────────────┐
│                      AI项目落地五阶段                         │
├──────────────────────────────────────────────────────────────┤
│  阶段一：业务调研  │  梳理完整流程，确认技术能力需求           │
├──────────────────────────────────────────────────────────────┤
│  阶段二：技术调研  │  调研业界方案，评估平台能力              │
├──────────────────────────────────────────────────────────────┤
│  阶段三：MVP落地   │  聚焦核心场景，快速验证假设              │
├──────────────────────────────────────────────────────────────┤
│  阶段四：数据优化  │  分析bad case，针对性调优               │
├──────────────────────────────────────────────────────────────┤
│  阶段五：灰度验证  │  双跑验证，逐步放量，风险可控            │
└──────────────────────────────────────────────────────────────┘
```

**为什么是这个顺序？**
- AI能力相对"黑盒"，很难一步到位做完美，必须先验证可行性
- 数据是模型的粮食，必须建立分析-优化的闭环
- 灰度双跑是控制AI方案负面影响的必要手段

## 六、方案详述：五阶段落地实践

### 6.1 阶段一：业务调研——明确"干什么、怎么干"

**核心任务：** 确认运营如何完成申诉判定，输出完整业务流程图和技术能力需求指标。

**业务流程梳理：**
从工单接收→材料收集→规则匹配→判定输出→结果反馈，全链路拆解每个步骤的输入、处理逻辑、输出。

**技术能力需求识别：**

| 能力需求 | 业务场景 | 占比评估 |
|---------|---------|---------|
| 视频理解与内容总结 | 处理用户上传的投诉视频 | 低（暂不覆盖） |
| 图片理解与内容总结 | 识别图片中的文字、场景、凭证 | 中 |
| 语音理解与内容总结 | 处理语音投诉、电话录音 | 中 |
| 判定规则检索与综合判定 | 基于规则库进行逻辑推理 | 高（核心能力） |

**关键决策：** 视频处理当时业内无成熟方案，且Q平台暂不支持，占比又低，MVP阶段暂不覆盖。

### 6.2 阶段二：技术调研——判定落地可行性

针对各项技术能力需求，评估可行性：

| 能力需求 | 技术实现路径 | 可行性结论 |
|---------|-------------|-----------|
| **视频理解** | 多模态大模型 | ❌ 当时业内无成熟方案，Q平台暂不支持 |
| **图片理解** | 多模态大模型（OCR+文本理解） | ✅ Q平台已接入支持图片识别的多模态大模型 |
| **语音理解** | ASR转文本 + 文本大模型 | ✅ Q平台可提供ASR，文本大模型已接入 |
| **规则检索与判定** | Dify知识库外挂 + qmoss Agent | ✅ Q平台提供Dify和Agent能力 |

**设计决策：**
- 优先攻克图片、语音和规则判定能力
- 视频场景留待技术成熟后再接入
- 基于平台已有能力，降低自主研发成本

### 6.3 阶段三：MVP落地——数据先行，快速验证

**数据准备（第一阶段）：**

"数据是模型的粮食"，MVP第一阶段优先完成数据准备：
- **数据整合：** 申诉数据、客服信息、订单日志分散于各系统，需要统一汇总
- **工具搭建：** C直采缺陷数据存储于C平台，无法接口同步，开发数据上传工具
- **质量保障：** 建立数据清洗流程，处理缺失值、异常值；构建数据标注规范

**逻辑实现（第二阶段）：**

系统架构设计：
```
申诉工单接入 → 材料解析（图片OCR/语音ASR） → 内容结构化 
    ↓
Dify知识库规则检索 → qmoss Agent综合推理 → 判定结论输出
    ↓
置信度评估 → 高置信自动判定 / 低置信人工复核
```

**提示词工程：**
- **角色定义：** 明确AI的审核员角色和判定标准
- **Few-shot示例：** 引入典型case的判定示例，提升模型对业务场景的理解
- **规则注入：** 将判定规则动态注入prompt，保证判定的可解释性
- **输出约束：** 规范输出格式（判定结果+理由+置信度），便于结果解析

### 6.4 阶段四：数据优化——分析驱动持续迭代

**bad case回流机制：**
- 人工复核结果自动回流到训练数据
- 定期分析模型误判原因（规则覆盖不足？语义理解偏差？）
- 针对性调整提示词和规则库

**A/B测试验证：**
- 不同提示词策略的效果对比
- 规则库版本迭代的效果评估
- 数据驱动，而非拍脑袋决策

### 6.5 阶段五：灰度验证——控制风险的关键

**为什么必须灰度？**

AI模型的"黑盒"特性决定了上线前必须充分验证。如果AI泛化能力弱，上线后出现较多错误判定：
- ❌ 令平台商户对平台能力产生质疑，影响平台形象
- ❌ 出现较多重复判定，大大增加平台投入成本
- ❌ 可能引发商户投诉，带来业务风险

**灰度双跑机制：**
1. AI判定结果与人工判定结果并行输出
2. 人工审核同学对AI判定结果进行复核
3. 监控AI判定的准确率、召回率、一致性指标
4. 逐步提升AI判定占比，从10%→50%→80%

**上线SOP：**
- 实时数据监控，异常指标自动熔断
- 完备的降级预案，确保人工兜底能力
- 判定结果可解释、可追溯、可干预

## 八、横向对比：与其他方案的对比

### 8.1 纯人工方案

| 维度 | 评价 |
|------|------|
| **优点** | 判定灵活，可处理复杂边界case |
| **缺点** | 效率低、成本高、一致性差、知识难沉淀 |
| **适用场景** | AI置信度低的复杂case |

### 8.2 纯规则引擎方案

| 维度 | 评价 |
|------|------|
| **优点** | 确定性高，可解释性强 |
| **缺点** | 规则维护成本高，难以覆盖所有场景，对模糊case处理能力弱 |
| **适用场景** | 边界清晰、规则明确的场景 |

### 8.3 纯AI大模型方案

| 维度 | 评价 |
|------|------|
| **优点** | 泛化能力强，可处理模糊场景 |
| **缺点** | 黑盒不可控，可能产生幻觉，缺乏可解释性 |
| **适用场景** | 容错率高、对准确性要求低的场景 |

### 8.4 我们的方案：规则+AI混合架构

| 维度 | 评价 |
|------|------|
| **优点** | 规则保证确定性，AI处理模糊场景，人机协作兜底 |
| **缺点** | 架构复杂度高，需要维护规则库和模型两套系统 |
| **适用场景** | 准确性要求高、规则部分可枚举、业务场景复杂的生产环境 |

**核心洞察：** 没有最好的方案，只有最适合的方案。对于"缺陷申诉判定"这种准确性要求高、规则部分明确、场景复杂的业务，"规则+AI+人工兜底"的混合架构是更务实的选择。

## 九、适用边界：诚实地说清楚什么适用、什么不适用

### 9.1 适用场景

✅ **规则部分可枚举：** 业务有明确的判定规则，即使规则很多
✅ **流程相对固定：** 申诉处理的流程标准化，变化频率低
✅ **数据可获取：** 判定所需的订单数据、客服记录等可系统获取
✅ **准确性要求高但可人机协作：** 允许AI处理高置信度case，低置信度case人工兜底

### 9.2 不适用场景

❌ **规则完全不可枚举：** 每个case都需要大量外部调研和主观判断
❌ **流程极度灵活：** 申诉处理流程经常变化，难以标准化
❌ **数据分散无法整合：** 判定所需数据分散在无法打通的系统中
❌ **容错率极低：** 任何错误判定都可能带来严重后果，且无法人工兜底

### 9.3 关键成功因素

1. **业务理解先于技术实现：** 深入理解业务流程和规则，比调模型更重要
2. **MVP思维：** 先验证核心假设，再逐步扩展，不追求完美
3. **数据驱动：** 建立分析-优化的闭环，持续迭代
4. **风险可控：** 灰度验证是必备环节，不能跳过

## 十、后续计划：持续迭代的方向

### 10.1 短期（3个月内）

- **提示词优化：** 基于生产数据持续优化prompt，提升准确率
- **覆盖场景扩展：** 将图片、语音理解能力推广至更多业务线
- **人工复核效率提升：** 优化复核界面，降低人工判定成本

### 10.2 中期（6个月内）

- **多模态能力提升：** 视频理解技术成熟后接入，覆盖全场景
- **智能推荐：** 基于历史数据，主动推荐相似case的判定依据
- **异常检测：** 自动识别异常申诉模式，辅助风险防控

### 10.3 长期（1年内）

- **通用智能审核平台：** 将AI判定能力平台化，支持其他业务线快速接入
- **Agentic Workflow：** 探索复杂场景的多轮推理和自主决策
- **模型自学习：** 建立case自动回流和模型自优化机制

## 十一、总结：一句话核心观点

**AI在业务系统的落地，核心矛盾不是"模型能力有多强"，而是"如何在工程化约束下，让人机协作创造最大价值"。**

技术只是手段，解决业务问题才是目的。与其追求"全自动AI判定"的理想，不如务实设计"AI处理标准化场景+人工兜底复杂case"的混合架构，在保证质量的前提下逐步提效。
